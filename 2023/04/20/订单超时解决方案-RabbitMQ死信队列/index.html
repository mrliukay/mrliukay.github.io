订单超时解决方案-RabbitMQ死信队列

RabbitMQ死信队列解决订单超时

摘要:

本文主要介绍利用分布式组件RabbitMQ解决订单超时的问题

1 死信以及TTL概念

死信，在官网中对应的单词为“Dead Letter”;

它是RabbitMQ中的一种消息机制

TTL：即消息的存活时间。RabbitMQ可以对队列和消息分别设置TTL，如果对队列设置，则队列中所有的消息都具有相同的过期时间。超过了这个时间，我们认为这个消息就死了，称之为死信。

1.1 队列中消息出现死信的几种情况

  1. 消息被否定确认,其实就是呗消费者拒绝.使用 channel.basicNack 或 channel.basicReject ，并且此时requeue 属性被设置为false;
  2. 消息在队列中存活的时间超过设置的TTL时间;
  3. 消息在队列的消息数量已经超过了最大的队列长度;(队列可以设置最大消息数量的属性,如果超出了这个范围那么多出来的就是死信).

1.2 如何配置死信队列

上面说到三种消息成为死信的情况,那么我们成为死信的消息将”何去何从”呢?

其实我们一样可以创建一个专门接收死信的队列来存放管理这些消息—死信队列

死信队列其实就是一个普通的队列;所以配置过程如下:

  1. 配置业务队列,配置业务交换机,并将业务队列绑定到交换机上;
  2. 配置业务队列的时候为业务队列设置属性,(TTL过期时间,死信交换机名称,可死信队列路由key)
  3. 配置死信队列,配置死信交换机(就是一个普通的交换机),为死信队列绑定死信交换机;

1.3 队列属性

上面说到为业务队列设置属性,以便队列中的消息变成死信之后通过死信交换机路由到死信队列,下面就是我们要配置的属性

  1. x-dead-letter-exchange 此属性设置死信交换机名称
  2. x-dead-letter-routing-key 此属性设置死信队列的routing key
  3. x-message-ttl 此属性设置队列中消息过期时间单位ms
  4. x-max-length 此属性设置队列的最大消息数量,超过这个值的消息将会变成死信

注意:

​ x-message-ttl属性设置的是队列中所有消息的过期时间,我们也可以在发送消息的时候单独设置每一个消息的过期时间;

​ 如果消息队列和消息都设置了过期时间,那么哪一个设置的时间小就按照哪一个.

2 代码实现

理论的东西整理结束了下面就可以直接上代码和演示截图了

下面是主要代码

2.1 配置死信队列死信交换机

/**
 * 死信队列交换机配置
 *
 * @author liukai
 * @since 2023/4/19 13:24
 */
@Configuration
public class DieExchangeQueueConfig {
    /**
     * 业务队列A
     *
     * @return {@link Queue }
     */
    @Bean
    public Queue busQueueA() {
        Map<String, Object> args = new HashMap<>();
        // 队列过期时间TTL,单位毫秒
        args.put("x-message-ttl", 5000);

        // x-dead-letter-exchange    这里声明当前队列绑定的死信交换机
        args.put("x-dead-letter-exchange", RabbitMqConstant.DIE_AB_TOPIC_EXCHANGE_NAME);

        // x-dead-letter-routing-key  这里声明当前队列的死信路由key
        args.put("x-dead-letter-routing-key", RabbitRoutingKeyEnum.DIE_QUEUE_A_ROUTING_KEY.getValue());
        return new Queue(RabbitMqConstant.BUS_QUEUE_A_NAME, true, false, false, args);
    }

    /**
     * 业务队列AB的交换机
     *
     * @return {@link TopicExchange }
     */
    @Bean
    public TopicExchange busAbTopicExchange() {
        return new TopicExchange(RabbitMqConstant.BUS_AB_TOPIC_EXCHANGE_NAME);
    }

    /**
     * 业务队列A绑定到业务AB交换机
     *
     * @return {@link Binding }
     */
    @Bean
    public Binding bindBusTopicExchangeA() {
        return BindingBuilder
                // 绑定队列
                .bind(busQueueA())
                // 到交换机
                .to(busAbTopicExchange())
                .with(RabbitRoutingKeyEnum.BUS_QUEUE_A_ROUTING_KEY.getValue());
    }

    /**
     * 死信队列A
     *
     * @return {@link Queue }
     */
    @Bean
    public Queue dieQueueA() {
        return new Queue(RabbitMqConstant.DIE_QUEUE_A_NAME);
    }

    /**
     * 死信交换机
     *
     * @return {@link TopicExchange }
     */
    @Bean
    public TopicExchange dieAbTopicExchange() {
        return new TopicExchange(RabbitMqConstant.DIE_AB_TOPIC_EXCHANGE_NAME);
    }

    /**
     * 死信队列A绑定死信交换机
     *
     * @return {@link Binding }
     */
    @Bean
    public Binding bindDieTopicExchangeA() {
        return BindingBuilder
                // 绑定队列
                .bind(dieQueueA())
                // 到交换机
                .to(dieAbTopicExchange())
                .with(RabbitRoutingKeyEnum.DIE_QUEUE_A_ROUTING_KEY.getValue());
    }
}

2.3 模拟生产方

@RestController
@RequiredArgsConstructor
@RequestMapping("/rabbitmq/test")
public class RabbitMqTestController {
    private final RabbitMqTestService rabbitMqTestService;
    
    @GetMapping("/send/message/topic/die/queue/a")
    public Map<String, Object> sendMessageTopicDieQueueA(String message) {
        return rabbitMqTestService.sendMessageTopicDieQueueA(message);
    }
}
@Service
@RequiredArgsConstructor
public class RabbitMqTestServiceImpl implements RabbitMqTestService {
    private final RabbitTemplate rabbitTemplate;
    
    @Override
    public Map<String, Object> sendMessageTopicDieQueueA(String message) {
        Map<String, Object> messageMap = getMessageMap(message);
        rabbitTemplate.convertAndSend(RabbitMqConstant.BUS_AB_TOPIC_EXCHANGE_NAME, RabbitRoutingKeyEnum.BUS_QUEUE_A_ROUTING_KEY.getValue(), messageMap);
        return messageMap;
    }
}

注意:

注意这种方式是正常发送消息到业务队列,我们在上面已经配置了业务队列的ttl(args.put(“x-message-ttl”, 5000);)所以这里面的消息如果超过可这个时间没有被消费者消费就会进入死信队列,我们消费方就会监听到死信队列中的消息,

此外我们还可以单独对消息设置ttl这样如果该消息的ttl如果比对列的小就会按照消息的ttl走

对消息单独设置TTL的代码如下:

@Override
    public Map<String, Object> sendMessageTopicDieQueueAttlmes(String message) {
        Map<String, Object> messageMap = getMessageMap(message);
        //给消息设置过期时间
        MessagePostProcessor messagePostProcessor = new MessagePostProcessor() {
            @Override
            public Message postProcessMessage(Message message) throws AmqpException {
                // 这里单位也是毫秒，但这里是字符串的形式
                message.getMessageProperties().setExpiration("3000");
                //还可以为消息设置编码
                message.getMessageProperties().setContentEncoding("UTF-8");
                return message;
            }
        };
        rabbitTemplate.convertAndSend(RabbitMqConstant.BUS_AB_TOPIC_EXCHANGE_NAME, RabbitRoutingKeyEnum.BUS_QUEUE_A_ROUTING_KEY.getValue(), messageMap, messagePostProcessor);
        return messageMap;
    }
